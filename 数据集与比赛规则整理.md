### 2.1 基本规模


这说明货源数据覆盖 2026 年 3 月整个月，是一个随时间动态上架和下架的货源池。

每一行是一条 JSON，例如：

```json

  "cargo_id": "1",
  "create_time": "2026-03-01 00:41:30",
  "remove_time": "2026-03-01 08:56:39",
  "cargo_name": "机械设备",
  "start": {
    "city": "广东省江门市江海区",
    "lat": 22.67,
    "lng": 113.15
  },
  "end": {
    "city": "广东省广州市白云区",
    "lat": 23.3,
    "lng": 113.34
  },
  "load_time": [
    "2026-03-01 08:00:00",
    "2026-03-01 09:00:00"
  ],
  "cost_time_minutes": 395,
  "price": 20301.15,
  "truck_length": [
    "3.8米",
    "4.2米"
  ]
}
```

### 2.2 字段说明

| 字段 | 含义 | 对 Agent 的影响 |
|---|---|---|
| `cargo_id` | 货源唯一 ID | `take_order` 时必须传这个 ID |
| `create_time` | 货源上架时间 | 当前仿真时间早于它时不可见 |
| `remove_time` | 货源下架时间 | 当前仿真时间超过后会从在线池消失 |
| `cargo_name` | 货物品类 | 很多司机偏好会禁接或尽量不接某些品类 |
| `start` | 装货点，含城市、经纬度 | 影响赴装货点空驶距离、耗时和成本 |
| `end` | 卸货点，含城市、经纬度 | 接单完成后司机位置会变到这里 |
| `load_time` | 装货时间窗 | 到早会等待，到晚会接单失败 |
| `cost_time_minutes` | 干线运输耗时，单位分钟 | 接单成功后推进的主要时间 |
| `price` | 货源价格 | 原始数据单位按分处理，收益脚本转成元 |
| `truck_length` | 适配车长列表 | 当前 Demo 本地评测未强校验，但正式策略应考虑 |

需要特别注意：

- 原始 `price` 字段按“分”理解；
- 收益脚本中会按 `price / 100` 换算为元；
- `query_cargo` 返回给 Agent 的候选货源中，价格已经被转换成元。

## 4. 司机偏好规则

`preferences` 是本赛题的核心难点之一。

它不是简单的结构化字段，而是自然语言规则，例如：

- 每天必须连续休息多少小时；
- 某些时间段不能接单或空驶；
- 某些货物品类不能接；
- 每月必须有几天不接单；
- 必须到某个地点；
- 不能进入某个区域；
- 某个特殊货源必须接。

官方文档提醒，选手侧不应在决策代码中为某个 `driver_id` 写死规则。更合理的做法是：

1. 从 `get_driver_status(driver_id)` 获取当前可见 `preferences`；
2. 将自然语言偏好理解或解析成结构化约束；
3. 决策时根据这些约束过滤候选动作和计算风险罚分。

### 4.1 D001

偏好：

1. 每天至少连续休息 8 小时；
2. 不接 `化工塑料`、`煤炭矿产`；
3. 车辆始终在深圳范围内：

```text
纬度 22.42 - 22.89
经度 113.74 - 114.66
```

### 4.2 D002

偏好：

1. 自然月内至少 4 个整天不接单；
2. 不接 `蔬菜`；
3. 每天至少连续休息 4 小时。

### 4.3 D003

偏好：

1. 月度空驶总里程不得超过 100 km，超出部分按公里罚；
2. 不得进入以 `(23.30, 113.52)` 为圆心、半径 20 km 的区域；
3. 每天凌晨 2 点到 5 点不接单、不空驶。

### 4.4 D004

偏好：

1. 如果当天接单，首单开工不得晚于当天 12:00；
2. 同一天接单不得超过 3 单，超出按单罚；
3. 每天 12:00-13:00 不接单、不空驶。

### 4.5 D005

偏好：

1. 单笔装货点到卸货点距离不得超过 100 km；
2. 接单后赴装货点空驶距离不得超过 90 km；
3. 每晚 23:00 到次日 06:00 不接单、不空驶。

### 4.6 D006

偏好：

1. 每天至少连续停车休息 5 小时；
2. 不接 `鲜活水产品`；
3. 单笔装货点到卸货点距离不得超过 150 km；
4. 自然月内至少 2 个整天不接单、不空驶。

### 4.7 D007

偏好：

1. 每天 23:00 到次日 04:00 不接单、不空驶；
2. 不接 `机械设备`；
3. 单笔装卸距离不得超过 180 km；
4. 自然月内至少放空一整天不接单。

### 4.8 D008

偏好：

1. 自然月内至少 2 天完全休息，不接单也不空驶；
2. 工作日每天连续停车休息 4 小时；
3. 尽量不拉 `食品饮料`；
4. 接单后赴装货点空驶距离不得超过 50 km。

### 4.9 D009

偏好：

1. 有一个特殊熟货源 `cargo_id = 240646`，不接会罚 10000；
2. 每天 23:00 前须回到家附近 `(23.12, 113.28)` 1 km 内，并且 23:00 到次日 08:00 不接单、不空驶；
3. 不接 `快递快运搬家`。

### 4.10 D010

偏好：

1. 2026-03-10 10:00 起有家事任务：
   - 先到 `(23.21, 113.37)` 接配偶；
   - 原地停留不少于 10 分钟；
   - 再回老家 `(23.19, 113.36)`；
   - 2026-03-10 22:00 前到家；
   - 到家后至少待到 2026-03-13 22:00；
   - 迟到或不在家按分钟罚，且可能额外罚 9000；
2. 自然月内至少 5 个不同自然日到过 `(23.13, 113.26)` 1 km 内；
3. 每天连续停车休息至少 3 小时；
4. 尽量不拉 `服饰纺织皮革`。

## 8. `demo/agent/`：选手主要修改位置

### 8.1 `demo/agent/model_decision_service.py`

这是当前示例 Agent 的核心决策入口。

核心类：

```python
class ModelDecisionService:
    def decide(self, driver_id: str) -> dict[str, Any]:
```

当前示例流程：

1. 调用 `get_driver_status(driver_id)` 获取司机状态；
2. 用当前经纬度调用 `query_cargo` 查询附近货源；
3. 构造 prompt；
4. 调用模型；
5. 解析模型输出动作。

支持三种动作格式：

```json
{"action": "take_order", "params": {"cargo_id": "..."}}
```

```json
{"action": "reposition", "params": {"latitude": 23.1, "longitude": 113.2}}
```

```json
{"action": "wait", "params": {"duration_minutes": 60}}
```

当前示例 Agent 比较弱，主要问题：

- 只取前 20 个候选货源；
- 没有把 `preferences` 传给模型；
- 没有把 `cargo_name` 放入候选信息；
- 没有系统化过滤偏好风险；
- 没有长期状态管理；
- 每步都调用模型，token 和时间成本高；
- 没有针对每日休息、夜间禁行、回家、特殊任务做规划。

后续算法设计的重点应该是改造这个文件。

## 9. `demo/simkit/`：仿真核心规则层

### 9.1 `demo/simkit/ports.py`

定义 Agent 和仿真环境之间的接口协议。

关键接口：

```python
def get_driver_status(self, driver_id: str) -> dict[str, Any]
def query_cargo(self, driver_id: str, latitude: float, longitude: float) -> dict[str, Any]
def query_decision_history(self, driver_id: str, step: int) -> dict[str, Any]
def model_chat_completion(self, payload: dict[str, Any]) -> dict[str, Any]
```

你的 Agent 应通过这些接口获取信息，而不是直接读数据文件。

### 9.2 `demo/simkit/cargo_repository.py`

货源仓库。

作用：

- 启动时加载 `cargo_dataset.jsonl`；
- 根据当前仿真时间维护两个池：
  - `pending`：未上架货源；
  - `online`：已上架且未过期货源；
- 查询时返回某个经纬度附近最近的 K 条在线货源。

这说明货源池是动态变化的，而不是静态列表。

### 9.3 `demo/simkit/driver_state_manager.py`

司机状态管理器。

作用：

- 加载 `drivers.json`；
- 维护当前司机位置；
- 维护当前仿真时间；
- 维护已完成订单数；
- 根据当前墙钟时间过滤可见偏好。

这说明某些偏好只会在特定时间窗口内通过接口返回。

### 9.4 `demo/simkit/simulation_actions.py`

三类动作和核心物理规则都在这里。

关键函数：

```python
query_cargo(...)
take_order(...)
wait(...)
reposition(...)
```

重要规则：

1. 距离用 Haversine 大圆距离；
2. 空驶耗时为：

```text
max(1, ceil(distance_km / speed_km_per_hour * 60))
```

3. `reposition` 即使目标和当前位置相同，也至少消耗 1 分钟；
4. `take_order` 时，如果已经在装货点附近，接单前空驶时间可以是 0；
5. `take_order` 会先把货源从在线池移除；
6. 如果到达装货点晚于装货窗结束，接单失败；
7. 如果提前到装货窗，会等待到装货窗开始；
8. 如果订单完成时间超过仿真 horizon，可能不计收入。

## 12. 比赛规则

### 12.1 任务本质

这是一个连续决策问题，不是单点预测问题。

Agent 每一步需要基于当前：

- 司机位置；
- 当前仿真时间；
- 可见货源；
- 司机偏好；
- 历史动作；
- 长期收益目标；

在三类动作中选一个：

```text
take_order
wait
reposition
```

目标是整个仿真周期的综合收益最大，而不是单笔订单收益最大。

### 12.2 动作 1：接单 `take_order`

格式：

```json
{
  "action": "take_order",
  "params": {
    "cargo_id": "123"
  }
}
```

执行过程：

1. 检查货源是否还在线；
2. 从当前位置空驶到装货点；
3. 如果早于装货时间窗，等待；
4. 如果晚于装货时间窗，失败；
5. 成功后推进 `cost_time_minutes`；
6. 司机位置更新到卸货点。

### 12.3 动作 2：休息 `wait`

格式：

```json
{
  "action": "wait",
  "params": {
    "duration_minutes": 300
  }
}
```

执行过程：

- 位置不变；
- 时间推进指定分钟数。

### 12.4 动作 3：空驶 `reposition`

格式：

```json
{
  "action": "reposition",
  "params": {
    "latitude": 23.1,
    "longitude": 113.2
  }
}
```

执行过程：

- 计算当前位置到目标点距离；
- 按速度换算耗时；
- 更新时间和位置。

## 13. 仿真时间、距离、成本规则

### 13.1 时间

仿真时间单位是分钟，起点是：

```text
2026-03-01 00:00:00
```

例如：

```text
simulation_progress_minutes = 60
```

表示：

```text
2026-03-01 01:00:00
```

### 13.2 距离

任意两点之间的距离使用 Haversine 大圆距离，单位为 km。

### 13.3 空驶速度

默认配置：

```text
60 km/h
```

空驶耗时：

```text
minutes = max(1, ceil(distance_km / speed_km_per_hour * 60))
```

### 13.4 查询货源也会消耗时间

`query_cargo` 不改变位置，但会消耗浏览时间：

```text
scan_minutes = ceil(返回货源条数 / cargo_view_batch_size)
```

Demo 默认：

```text
cargo_view_batch_size = 10
nearest_cargo_limit = 100
```

所以一次最多返回 100 条时，会消耗：

```text
ceil(100 / 10) = 10 分钟
```

这点非常重要。频繁查询会推进仿真时间，可能导致：

- 货源过期；
- 错过装货窗；
- 进入夜间禁行时间；
- 压缩有效运营时间；
- 影响每日休息或回家规则。

## 14. 评测规则

### 14.1 评测执行顺序

官方文档给出的流程是：

1. 运行仿真：

```bash
demo/server/main.py
```

2. 生成动作日志：

```text
demo/results/actions_202603_*.jsonl
```

3. 运行收益脚本：

```bash
demo/calc_monthly_income.py
```

4. 生成评分结果：

```text
demo/results/monthly_income_202603.json
```

### 14.2 动作日志

每个司机会有一个动作日志文件：

```text
actions_202603_D001_*.jsonl
actions_202603_D002_*.jsonl
...
```

每一行是一条 step 记录，包括：

- `step`；
- `driver_id`；
- `step_elapsed_minutes`；
- `query_scan_cost_minutes`；
- `action_exec_cost_minutes`；
- `position_before`；
- `position_after`；
- `simulation_end_time`；
- `action`；
- `token_usage`；
- `result`。

### 14.3 时间一致性校验

收益脚本会检查：

```text
step_elapsed_minutes = query_scan_cost_minutes + action_exec_cost_minutes
```

并且：

```text
本步结束时间 - 上步结束时间 = step_elapsed_minutes
```

### 14.4 位置一致性校验

- `wait`：前后位置必须不变；
- `reposition`：结束位置必须等于目标位置；
- `take_order` 成功：结束位置必须等于货源终点。

### 14.5 接单时间合法性

成功接单时，接单动作开始时间必须在货源有效期内：

```text
create_minutes <= action_start_minutes <= remove_minutes
```

### 14.6 接单耗时合法性

成功接单耗时必须等于：

```text
赴装货点空驶耗时 + 装货窗等待耗时 + cost_time_minutes
```

### 14.7 收益计算公式

对每个司机：

```text
gross_income = 成功接单货源价格总和
distance_km = 空驶里程 + 接单赴装货点空驶里程 + 成功订单干线里程
cost = distance_km * cost_per_km
net_income = gross_income - cost - preference_penalty
```

其中：

- `gross_income` 按 `cargo_id` 回查原始货源价格；
- 原始价格会除以 100 转为元；
- `distance_km` 包含空驶和干线里程；
- `cost_per_km` 来自司机数据；
- 当前司机成本大多是 1.5 元/km；
- `preference_penalty` 来自偏好规则校验。

## 15. 特别需要注意的地方

### 15.1 Agent 决策代码禁止直接读取原始数据

这是最重要的规则之一。

禁止直接读取：

```text
demo/server/data/cargo_dataset.jsonl
demo/server/data/drivers.json
```

允许通过接口获取：

```python
get_driver_status(driver_id)
query_cargo(driver_id, latitude, longitude)
query_decision_history(driver_id, step)
model_chat_completion(payload)
```

现在可以读取数据文件是为了分析和理解规则；后续写正式 Agent 时不能这样做。

### 15.2 不要硬编码司机 ID 规则

虽然公开数据里有 D001-D010，但不建议写类似：

```python
if driver_id == "D009":
    ...
```

原因：

- 初赛后续数据可能更新；
- 复赛司机集合可能隐藏；
- 偏好文本可能变化；
- 硬编码容易在正式评测失效。

更合理做法：

1. 从 `get_driver_status` 获取当前可见 `preferences`；
2. 用 LLM 或规则解析器将偏好转成结构化约束；
3. 策略执行时遵守这些约束。

### 15.3 当前示例 Agent 没有把偏好传给模型

当前 `model_decision_service.py` 虽然获取了 `status`，但构造 prompt 时只保留了：

```text
current_lat
current_lng
truck_length
completed_order_count
```

它没有把 `preferences` 放进去，也没有放 `cargo_name`。

这会导致模型无法知道很多偏好规则。

后续第一步改造应包括：

- 把 `simulation_wall_time` 放入上下文；
- 把 `preferences` 放入上下文；
- 把候选货源的 `cargo_name`、`truck_length`、`load_time`、`start/end`、距离、价格都放进去；
- 最好不要完全依赖模型自由判断，而是有程序化打分和约束过滤。

### 15.4 `query_cargo` 会推进时间

这点很容易忽略。

示例 Agent 每一步都先查询货源。如果返回 100 条，就会消耗 10 分钟。查询 6 次就是 1 小时。

如果无节制地查询多个城市，会：

- 错过装货窗口；
- 让当前候选货源失效；
- 压缩一天内可接单时间；
- 影响夜间、午休、休息、回家等偏好。

因此 Agent 设计里需要限制查询次数，并选择有价值的查询点。

### 15.5 查询坐标可以不是当前位置

接口是：

```python
query_cargo(driver_id, latitude, longitude)
```

并不强制只能查当前位置。

这意味着可以查询某些 hub 附近的货源，例如：

- 广州白云；
- 佛山南海；
- 深圳龙岗；
- 惠州惠城；
- 东莞热点镇区。

但查询会耗时，所以不能全图乱扫。

### 15.6 接单前要检查多个时间

一个货源是否值得接，不只看价格。

至少要估算：

```text
当前时间
+ 查询耗时
+ 赴装货点空驶耗时
+ 等待装货窗时间
+ cost_time_minutes
```

还要检查：

- 是否已经超过 `remove_time`；
- 到达装货点是否晚于 `load_time` 结束；
- 完单是否超过仿真周期；
- 是否撞上司机禁行、休息、回家规则。

### 15.7 月末订单可能不计收入但仍消耗成本

如果预计完成时间超过仿真 horizon，订单可能不计收入。

所以临近仿真结束时，长单风险很高：

- 可能没有收入；
- 但距离成本仍可能被计入；
- 还可能破坏偏好规则。

### 15.8 `wait` 和 `reposition` 的区别很重要

如果想休息，应该用：

```json
{"action": "wait", "params": {"duration_minutes": 300}}
```

不要用原地 `reposition`。

原因：

- `wait` 是休息，位置不变；
- `reposition` 是空驶行为，即使原地也会至少消耗 1 分钟；
- 很多偏好明确禁止“空驶”或要求“连续停车休息”。

### 15.9 偏好罚分可能比单笔收益大很多

例如：

- D009 特殊熟货不接，罚 10000；
- D010 家事任务失败，可能罚 9000 + 按分钟罚；
- 某些司机夜间规则累计罚分很高；
- 某些无限封顶规则可能持续扣分。

所以不能只做局部收益最大化。

一个 500 元高收益订单，如果导致 9000 元偏好罚分，就是灾难。

### 15.10 本地 Demo 配置默认只跑 1 天

`config.example.json` 中：

```json
"simulation_duration_days": 1
```

这只是 Demo 快速跑通用的配置。

比赛描述是月度连续仿真，正式评测应以赛方配置为准。

本地设计策略时，建议至少手动改成更长时间测试，例如 7 天或 30 天，观察长期效果。

## 16. 对 Agent 算法设计的启发

建议不要把 Agent 做成“每步让大模型随便选一个动作”。更稳妥的方向是构建一个有结构的决策系统。

推荐架构：

```text
状态层
偏好理解层
候选生成层
候选打分层
动作执行层
```

### 16.1 状态层

维护：

- 当前司机；
- 当前仿真分钟；
- 当前墙钟时间；
- 当前经纬度；
- 已完成订单数；
- 今日已接单数；
- 今日休息区间；
- 月度休息天数；
- 空驶累计里程；
- 是否已完成特殊任务；
- 历史动作。

数据来源：

```python
get_driver_status(driver_id)
query_decision_history(driver_id, step)
```

也可以在 Agent 实例内部维护内存，但要确保和历史记录一致。

### 16.2 偏好理解层

把自然语言 `preferences` 转成结构化约束，例如：

```json
{
  "forbidden_cargo_names": ["蔬菜"],
  "daily_rest_minutes": 240,
  "night_no_drive_windows": [["23:00", "06:00"]],
  "max_pickup_deadhead_km": 50,
  "monthly_off_days": 2
}
```

建议：

- 高罚分、无封顶规则当硬约束；
- 低罚分、有封顶规则可作为软约束；
- 特殊事件，例如 D009、D010，需要单独形成任务计划。

### 16.3 候选生成层

每步可以生成候选动作：

1. 当前点附近货源；
2. 重要 hub 附近货源；
3. 等待到某个时间点；
4. 空驶到某个 hub；
5. 回家、到指定点、休息。

不要每步全图搜索。可以维护一组高价值 hub：

```text
广州白云
佛山南海
佛山顺德
深圳龙岗
深圳宝安
惠州惠城
东莞热点区域
```

### 16.4 候选打分层

对每个候选订单计算近似价值：

```text
score =
  订单收入
  - 赴装货点空驶成本
  - 干线距离成本
  - 等待时间机会成本
  - 偏好风险罚分
  - 时间窗风险
  - 月末超时风险
```

也可以计算单位时间收益：

```text
profit_per_hour = expected_net_profit / expected_total_minutes
```

但不能只看 `profit_per_hour`，还要考虑订单结束位置是否进入高货源区域。

### 16.5 动作执行层

输出动作前做最后校验：

- action 是否合法；
- cargo_id 是否来自当前查询结果；
- 是否会违反硬偏好；
- 是否会超过装货窗；
- 是否会超过仿真 horizon；
- wait 时间是否为正整数；
- 经纬度是否合理。

如果没有好订单：

1. 优先满足休息或特殊任务；
2. 然后考虑等待；
3. 再考虑空驶到高价值 hub。

## 17. 推荐算法路线

### 17.1 阶段 1：构建强基线，不依赖复杂 LLM

每步：

1. 获取司机状态和偏好；
2. 查询当前位置附近货源；
3. 过滤明显不合法或高罚分订单：
   - 品类禁接；
   - 装货窗赶不上；
   - 超出距离限制；
   - 夜间禁行冲突；
   - 月末无法完成；
4. 对剩余订单按预期净收益排序；
5. 选择最优订单；
6. 没有合适订单则 `wait` 或 `reposition`。

这会比当前 Demo 强很多。

### 17.2 阶段 2：加入偏好状态机

为每个司机维护：

- 今日休息是否完成；
- 今日禁行窗口；
- 今日接单数；
- 月度 off-day 计数；
- 空驶累计；
- 特殊任务完成状态。

这样可以避免大量偏好罚分。

### 17.3 阶段 3：加入区域策略

基于数据热点，做区域选择：

- 如果当前位置附近货源少或收益低，查询几个 hub；
- 如果某个 hub 有明显更好机会，空驶过去；
- 否则等待一段时间再查。

### 17.4 阶段 4：谨慎使用 LLM

LLM 更适合做：

- 解析自然语言偏好；
- 解释复杂临时任务；
- 在候选动作分数接近时辅助决策。

不建议每步都让 LLM 从原始货源列表里自由选择。这样成本高、不可控、容易格式错误。
